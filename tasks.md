# Декомпозиция задачи

## 1. Подготовка и базовая конфигурация
- Проверить текущую структуру Symfony-проекта и точки расширения (`Controller`, `Message`, `MessageHandler`, `Service`).
- Вынести параметры RabbitMQ в `.env`/`.env.test`/`.env.prod`.
- Настроить `messenger.yaml` для RabbitMQ транспорта.

Критерий готовности:
- Проект поднимается локально.
- Конфиг Messenger валиден и использует переменные окружения.

## 2. Контракт входного API и DTO
- Описать формат входного payload для `POST /api/logs/ingest`.
- Создать DTO (или equivalent) для batch и отдельного лога.
- Зафиксировать обязательные поля: `timestamp`, `level`, `service`, `message`.

Критерий готовности:
- Контракт явно определен в коде.
- Поддерживается массив `logs`.

## 3. Валидация входных данных
- Реализовать валидатор структуры запроса.
- Проверить лимит: не более 1000 логов в batch.
- Проверить обязательные поля и формат `timestamp`.
- Подготовить формат ошибок валидации для ответа `400`.

Критерий готовности:
- Некорректные payload возвращают `400` с понятным описанием.
- Валидный payload проходит в обработку.

## 4. Реализация API endpoint
- Реализовать `POST /api/logs/ingest`.
- Сгенерировать уникальный `batch_id` для каждого запроса.
- Вернуть `202 Accepted` с `status`, `batch_id`, `logs_count`.

Критерий готовности:
- Endpoint соответствует контракту ответа из ТЗ.
- `logs_count` совпадает с фактическим размером batch.

## 5. Публикация в RabbitMQ через Symfony Messenger
- Создать message-класс для события ingest (по логу или по batch).
- Добавить метаданные: `batch_id`, `published_at`, `retry_count`.
- Настроить публикацию в очередь `logs.ingest` (direct, durable/persistent).
- Определить стратегию: одно сообщение на лог или один batch (зафиксировать выбор в README).

Критерий готовности:
- Сообщения отправляются в нужную очередь.
- Сообщения persistent.

## 6. Обработка ошибок и отказоустойчивость
- Обработать исключения при публикации в брокер.
- Настроить корректные коды ответа и безопасные сообщения об ошибках.
- Проверить, что бизнес-валидация и инфраструктурные ошибки разделены.

Критерий готовности:
- Ошибки валидации -> `400`, успешный прием -> `202`.
- Ошибки публикации корректно логируются/обрабатываются.

## 7. Unit тесты
- Написать unit-тесты для валидатора логов.
- Покрыть позитивные и негативные сценарии (обязательные поля, лимит 1000, формат timestamp).

Критерий готовности:
- Unit-тесты проходят стабильно.
- Основные ветки валидации покрыты.

## 8. Integration тесты
- Написать integration-тесты для `POST /api/logs/ingest`.
- Использовать mock/stub RabbitMQ (или test transport Messenger).
- Проверить сценарии: `202` на валидном payload, `400` на невалидном.

Критерий готовности:
- Integration-тесты подтверждают контракт API и факт dispatch/publish.

## 9. Документация и запуск
- Обновить `README.md`: запуск, переменные окружения, описание endpoint.
- Добавить примеры `curl` для успешного и ошибочного запроса.

Критерий готовности:
- По README новый разработчик может поднять сервис и отправить тестовый запрос.

## 10. Финальная проверка (Definition of Done)
- Все тесты проходят.
- Endpoint соответствует ТЗ (контракт, коды, лимиты, batch_id).
